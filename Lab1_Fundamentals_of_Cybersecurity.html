1 Overview

The objective of this lab is to practise the core ideas from Topic 1: what cybersecurity protects, the CIA triad (Confidentiality, Integrity, Availability), shared responsibility, risk ranking, attack-surface mapping, asset valuation, attack trees, and a first look at real web-application vulnerabilities.

After completing this lab, you should be able to:

  • Explain what cybersecurity protects and why it is a shared responsibility.
  • Classify security incidents against the CIA triad and justify each decision.
  • Investigate system logs and identify suspicious activity such as impossible travel.
  • Calculate and rank risk using likelihood and impact, and explain why likelihood alone is not priority.
  • Map network, software, and human attack surfaces and propose matching controls.
  • Identify asset types (data, software/service, people) and explain their value.
  • Build a simple attack tree with prevention and detection points.
  • Demonstrate and then patch SQL injection, data exposure, privilege abuse, and weak session-cookie vulnerabilities in a live web app.

2 Part A: Concept Warm-Up

Answer each question in your own words first, then reveal the suggested answer to compare.

A1. What does cybersecurity protect?
Show suggested answer

Cybersecurity protects systems, networks, data, services, applications, devices, and people from unauthorized access, misuse, disruption, or damage.

A2. Give examples of a confidentiality breach.
Show suggested answer

Leaked passwords, exposed student records, or stolen credit-card data.

A3. Give examples of an integrity breach.
Show suggested answer

An attacker changes exam marks, modifies a bank-transfer amount, or changes a delivery address.

A4. Give examples of an availability breach.
Show suggested answer

A DDoS attack, ransomware encryption, or a portal outage during registration.

A5. Why is cybersecurity a shared responsibility?
Show suggested answer

Because users, developers, administrators, managers, and vendors can all create or reduce risk.

9 Part G: Vulnerabilities and Security Measure Practice

This part is hands-on. You are provided with a deliberately vulnerable web application (a Campus User Portal) in the Lab1_Practice folder. You will exploit four weaknesses, then patch them and verify the fixes.

Authorized practice only

The techniques below are for this local, self-contained lab app only. Never run them against systems you do not own or are not explicitly authorized to test.

Lab Setup

  1. Open PowerShell.
  2. Navigate to the Lab1_Practice directory: 
    cd path\to\Lab1_Practice
    The prompt should end with ...\Lab1_Practice>.
  3. Start the web application: 
    .\start_lab.ps1
    Wait until initialization finishes.
  4. Open a browser and visit: 
    http://localhost:8090
    If it started successfully, the login page is displayed.
If PowerShell blocks the script (execution policy)

(a) What sensitive data is exposed? (b) Why is this dangerous?

Show suggested answer

(a) Plaintext passwords and other users' records are exposed.

(b) A normal user can discover the admin password and escalate privileges.

Question 3: Admin Privilege Abuse

Use the exposed admin credential to log in as admin.

(a) What actions can the admin perform? (b) Which CIA component is affected if an attacker gains admin access?

Show suggested answer

(a) The admin can edit and delete user records.

(b) Integrity is affected because records can be modified or deleted. Confidentiality is also affected because the admin can view user data.

Question 4: Cookie Security

Demonstrate session hijacking by copying a valid session cookie into another browser session.

  1. Log in using the admin account.
  2. Open Developer Tools with F12 and go to Application → Storage → Cookies → http://localhost:8090.
  3. Locate the cookie named sid and copy its value.
  4. Open a new Incognito / Private window and visit http://localhost:8090.
  5. Open Developer Tools (F12) and switch to the Console tab. Enter (replace PASTE_VALUE_HERE with the copied value): 
    document.cookie = "sid=PASTE_VALUE_HERE; Path=/";
  6. Refresh the page: 
    location.reload();

What result do you see?

Show suggested answer

The browser may gain access to the existing authenticated session without requiring a login. This demonstrates the risk of session hijacking when session identifiers are exposed or stolen. An attacker who obtains a valid session cookie may be able to impersonate a legitimate user and access protected resources without knowing the user's password.
